## 中期规划（与 TODO 最契合）

> 若时间有限，建议按此顺序：**SQLite 持久化 → 后台定时抓取 → 源健康度 + 去重**。

### [ ] 抓取结果写入本地轻量数据库（SQLite）

**背景**

当前文章数据仅保存在内存缓存（Flask-Caching `SimpleCache`）中，缓存过期后需重新并发抓取 200 个 RSS 源，首次加载耗时长、易卡顿，且无法保留历史文章。

**目标**

- 将每次抓取到的文章持久化到本地轻量数据库
- 页面/API 优先从数据库读取，减少重复抓取
- 保留一定时间范围内的历史记录，便于后续扩展搜索、统计等功能

**推荐方案（轻量）**

| 方案 | 说明 |
|------|------|
| **SQLite**（推荐） | 单文件、零配置、Python 内置支持，适合单机部署 |
| JSON/SQLite 混合 | 仅作过渡方案，数据量大时查询与去重不便 |

**建议表结构（草案）**

```
sources          — 信息源配置快照（可选，与 config.py 同步）
articles         — 文章：title, link, published_at, source_id, category, country, fetched_at
article dedup    — 以 link 或 link 哈希为唯一键，避免重复入库
fetch_logs       — 每次全量/增量抓取的时间、成功/失败源数量
```

**实现步骤**

1. 引入 SQLite（`sqlite3` 标准库或 SQLAlchemy 轻量封装）
2. 在 `fetch_feed` / `fetch_all_articles` 完成后，将结果 upsert 到 `articles` 表
3. 新增数据访问层：`get_articles(page, filters)` 从 DB 查询，替代直接读内存列表
4. `/api/articles` 改为读 DB；后台定时或启动时增量更新 RSS
5. 配置项：DB 路径（默认 `data/aggregator.db`）、历史保留天数、是否启动时自动抓取
6. （可选）抓取失败的信息源写入 `fetch_logs`，便于排查

**验收标准**

- [ ] 重启服务后无需等待全量 RSS 抓取即可浏览已有文章
- [ ] 同一文章链接不重复入库
- [ ] 筛选、分页在 DB 层完成，性能可接受（万级文章内秒开）
- [ ] 提供简单的 DB 初始化/迁移说明（README 或 `scripts/init_db.py`）

**相关文件**

- `app.py` — 抓取与 API 逻辑
- `config.py` — 信息源配置
- `templates/index.html` — 前端分页已就绪，后端换 DB 后接口可保持不变
